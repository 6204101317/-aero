ข้อสังเกตสำคัญ: หน้าจอหลายหน้า "ประกาศ" ตัวแปร WCF service client (เช่น n_loanClient loanService) ไว้ในโค้ดแต่ไม่เคยเรียกใช้งานจริง — เป็น boilerplate จาก template ที่ไม่ได้ implement ให้ครบ เป็น dead code ที่พบซ้ำในหลายหน้าจอ

ws_lon_clsmth_ctrl

ws_lon_clsmth.aspx.cs (110 บรรทัด)

สั่งปิดงวดบัญชีรายเดือนของระบบเงินกู้ พร้อมคำนวณดอกเบี้ยค้างชำระของสัญญาที่ค้างส่ง เป็นหน้า trigger งาน batch ล้วนๆ

Event handlers / key methods
  • Cls_Mth_Close() — ALTER TABLE lncontmaster ADD int_payment (schema patch inline แบบ try/catch เงียบ) แล้วส่ง batch
  • GetRealObjectApp — query amsecwins หา win_parameter ของหน้าจอปัจจุบัน
Service / data calls
  • <strong>WebUtil.runProcessingExtend(state, "LNCLOSEMONTH", ...)</strong> และ <strong>WebUtil.runProcessingExtend(state, "LNCALINTARRNEW", ..., "loan")</strong> — batch queue ทั้งคู่
Business rules / validation
  • ไม่มี validation input (ไม่เช็คปี/เดือนซ้ำ) — ปล่อยให้ batch process ตรวจสอบเอง
Notable: ไม่พบ hardcode coop_id

Event handlers / key methods
  • PostMemberNo/PostLoanContractNo — โหลดรายละเอียดสัญญา/รายการเคลื่อนไหว/หลักประกัน/การแบ่งชำระ

Service / data calls

Notable: coop "017001" เท่านั้นที่บันทึก audit log ลง lnauditloanlog ทุกครั้งที่แก้ไขสัญญา — audit trail พิเศษเฉพาะสหกรณ์นี้

ws_lon_keep_adjust_ctrl

ws_lon_keep_adjust.aspx.cs (143 บรรทัด)

ปรับปรุง/คำนวณใหม่รายการเรียกเก็บ (keeping) ของงวดที่ยังไม่ปิด เมื่อสัญญาเงินกู้เปลี่ยนแปลงระหว่างงวด (regenerate) — เชื่อมโยงกับโมดูล Keeping ที่วิเคราะห์ไว้ก่อนหน้านี้

Event handlers / key methods
  • WebSheetLoadBegin — หา recv_period ล่าสุดจาก kptempreceive มา pre-fill
  • PostReGenKeeping — หัวใจของหน้านี้: วนทุกแถว คำนวณดอกเบี้ย/เงินต้น/ยอดคงเหลือใหม่ เก็บ buffer ใน field ...NEW รอ SaveWebSheet เขียนจริง
  • SaveWebSheet — UPDATE kptempreceivedet (ตั้ง case_post='REGEN') + UPDATE lncontmaster (reset interest_return=0)
Service / data calls
  • wcf.NLoan.of_computeinterest — synchronous คำนวณดอกเบี้ยงวดใหม่ แล้ว direct SQL update — <strong>ไม่มี transaction wrapper (Sta) ต่างจาก ws_lon_auditloan</strong> ที่ห่อด้วย Sta ta ชัดเจน
Business rules / validation
  • ดอกเบี้ยกินเงินก่อนเงินต้นเสมอ (ถ้ายอดชำระเดิมน้อยกว่าดอกเบี้ยใหม่ ดอกเบี้ยกินหมด ไม่มีส่วนตัดต้น)
  • clamp เงินต้น/ยอดคงเหลือไม่ให้ติดลบ
  • ไม่มี validation ก่อน save (ไม่เช็คว่ากด regen ก่อนหรือยัง)
Notable: field case_post='REGEN' เป็น marker สำคัญสำหรับ traceability ของระบบ Keeping ว่าแถวไหนถูก regenerate จากหน้านี้ — ไม่มี hardcode coop_id ภายในไฟล์ (ความเป็น AERO-specific คือทั้งหน้าจอ ไม่ใช่ if-branch)
