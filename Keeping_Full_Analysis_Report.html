1. ภาพรวมและวิธีการ (Overview & Methodology)

โมดูล Keeping (เรียกเก็บเงิน) เป็นระบบหักเงินสมาชิกสหกรณ์ประจำงวด ครอบคลุมหุ้น เงินกู้ เงินฝาก เบี้ยประกัน ค่าแรกเข้า และเงินคืน โดยมีทั้งหน้าจอ ASP.NET (.NET WebForms) สำหรับตั้งค่า/ตรวจสอบ/สั่งงาน และเอนจินประมวลผลแบบ batch ฝั่ง PowerBuilder (PBProcess.exe) ที่ทำงานจริงเบื้องหลัง

ระบบมีซอร์สโค้ดอยู่ 2 โฟลเดอร์คือ CORE (เวอร์ชันฐาน) และ AERO (เวอร์ชันที่ถูกปรับแก้เพิ่มเติมสำหรับหน่วยงานนี้) จากทั้งหมด 37 รายการ (window_id) ภายใต้ application = keeping ใน amsecwins: 30 รายการอ่านจาก CORE และ 7 รายการอ่านจาก AERO (พบ 1 รายการ ws_dag_pay_mthdag_memberno ที่ระบุใน amsecwins แต่ไม่พบไฟล์จริงในโฟลเดอร์ทั้งสองฝั่ง — ดู Open Issues)

หมายเหตุ: runPostProcess (เรียกผ่าน WebUtil.runProcessingCappExtend ที่ ws_kep_check_kptp และ ws_dag_process_mthdag) ไม่ได้วิ่งผ่าน PBProcess.exe แต่วิ่งผ่าน AppProcessing.exe ซึ่งเป็นเอนจิน .NET คนละตัว

2. กฎการเลือกแหล่งข้อมูล (Source Resolution Rule)

• core_flag = 1 → อ่านจาก CORE

• core_flag = 0 หรือ 2 (อื่นๆ) → อ่านจาก AERO

อ้างอิงจากไฟล์ amsecwins (application = "keeping") — เมื่อ object ชื่อเดียวกันมีอยู่ทั้งในโฟลเดอร์ CORE และ AERO ให้ยึดกฎนี้เป็นแหล่งความจริง (source of truth)

3. แผนภาพ Swimlane ภาพรวมกระบวนการ

แบ่งกระบวนการออกเป็น 6 เฟส (คอลัมน์) และ 5 บทบาท/ชั้นระบบ (แถว) พร้อมระบุแหล่งที่มาของหน้าจอหลักด้วยสี — น้ำเงิน = CORE, ส้ม = AERO

5. สรุป PRD (Product Requirements)

เป้าหมาย (Objectives)

  • รวบรวมข้อกำหนด (requirements) ของทุกหน้าจอในโมดูล Keeping จากทั้งสองแหล่งโค้ดให้เป็นเอกสารอ้างอิงเดียว
  • กำหนดกฎการเลือกแหล่ง object ที่ชัดเจนและตรวจสอบย้อนกลับได้ (traceable) ตาม amsecwins.core_flag
  • เป็นฐานสำหรับทีมพัฒนาในการแก้ไข ทดสอบ หรือย้ายระบบ (migration) ในอนาคตโดยไม่ต้องไล่อ่านซอร์สโค้ดใหม่ทั้งหมด

ขอบเขต (Scope)

อยู่ในขอบเขต: window objects ทั้งหมด 37 รายการภายใต้ application = "keeping" ใน amsecwins (30 จาก CORE, 7 จาก AERO); การไหลของข้อมูลตั้งแต่ตั้งค่า → ยื่นคำขอ/นำเข้า → ตรวจสอบ/ปรับปรุง → ประมวลผลหลัก → กระบวนการย่อยที่เกี่ยวข้อง → หลังประมวลผล; จุดเชื่อมกับโมดูลเงินกู้ ประกัน เงินฝาก และตัวแทนหักหนี้ เฉพาะส่วนที่ทริกเกอร์จากรอบเก็บเงิน

นอกขอบเขต: Business logic ภายใน object PowerBuilder แบบ line-by-line ทั้งหมด; โมดูลอื่นที่ไม่ถูกอ้างอิงจากรอบเก็บเงินโดยตรง; รายการ ws_dag_pay_mthdag_memberno ที่ไม่พบไฟล์จริง

ข้อกำหนดที่ไม่ใช่เชิงฟังก์ชัน (Non-Functional Requirements)

  • Concurrency control: ป้องกันการสั่งประมวลผลซ้ำซ้อน โดยตรวจสอบ cmprocessing.runtime_status ก่อนเริ่มงานใหม่ทุกครั้ง
  • Auditability: ทุกรายการที่ตัดยอดแล้วต้องมีเลขที่เอกสารอ้างอิงผ่าน cmdocumentcontrol และเลขที่ใบเสร็จที่ไม่ซ้ำกัน
  • Data consistency: ค่าคงที่ควบคุมการประมวลผล (kpconstant) ต้องสอดคล้องกับค่าที่ batch engine ฝั่ง PowerBuilder อ่านไปใช้แบบ real-time
  • Multi-tenancy: ทุก transaction ต้องผูกกับ coop_control/coop_id เพื่อแยกข้อมูลรายสหกรณ์
  • Notification: ต้องรองรับการแจ้งเตือนผ่าน SMS สำหรับกรณีเฉพาะ (เช่น สหกรณ์ EGAT)

pckeeping.pbl เป็นไฟล์ compiled ของ PowerBuilder (p-code) ไม่ใช่ source code แบบ text ธรรมดา การอ่านทำโดย extract สตริงที่ฝังอยู่ในไฟล์แบบ UTF-16LE (strings -e l) ซึ่งสามารถอ่านชื่อ object, ชื่อฟังก์ชัน, และคำสั่ง SQL ที่ฝังในโค้ดได้ แต่ไม่สามารถอ่าน logic แบบ step-by-step ทั้งหมดได้ครบถ้วนเหมือน source code ต้นฉบับ (.srw/.sru)

คำอธิบายของแต่ละฟังก์ชันในเอกสารนี้อ้างอิงจาก 3 แหล่งประกอบกัน: (1) รูปแบบการตั้งชื่อฟังก์ชันตามธรรมเนียม PowerBuilder/GCOOP, (2) บริบทจาก choose case ใน dispatcher และลำดับการเรียกที่พบในสตริง, และ (3) ความสอดคล้องกับ workflow ที่ทราบจาก PRD ของโมดูล Keeping ผู้ใช้ที่ต้องการรายละเอียด logic ระดับบรรทัดควรเปิดไฟล์ต้นฉบับด้วย PowerBuilder IDE (Object Browser)
