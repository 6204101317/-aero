PRD: ระบบเงินกู้ (Loan System) — วิเคราะห์ครบวงจร 3 ชั้น (ASP.NET / PBProcess / PBService125)

เอกสารนี้ครอบคลุม: ชั้น UI/Validation (ASP.NET WebForms C#, แยก CORE และ AERO-extend), ชั้นประมวลผล batch แบบ asynchronous (PBProcess pcloan.pbl), และชั้น service ที่เรียกแบบ synchronous ผ่าน WCF (PBService125 loansrv.pbl และ lncoopsrv.pbl)

หลักการอ่านหลักฐาน (Evidence legend):

  • ✅ ยืนยันจาก source จริง — อ่านโค้ดจริงจากไฟล์ .aspx.cs (uncompiled, อ่านได้เต็ม) หรือ string-extraction จากไฟล์ .pbl (compiled binary, ใช้ strings -e l แกะ UTF-16LE)
  • 🟡 อนุมานจากชื่อ/บริบท — สำหรับหน้าจอ/ฟังก์ชันที่มีจำนวนมาก ไม่ได้อ่าน source ทีละบรรทัดทั้งหมด แต่อนุมานหน้าที่จาก naming convention และ pattern ที่พบซ้ำในกลุ่มเดียวกัน

1. ภาพรวม (Overview)

ระบบเงินกู้ของ GCOOP ประกอบด้วย 3 ชั้นการทำงานที่แยกจากกันทางกายภาพแต่เชื่อมกันด้วย 2 กลไกหลัก คือ การเรียก WCF แบบ synchronous (สำหรับธุรกรรมที่ต้องรอผลทันที) และ คิวงาน batch แบบ asynchronous ผ่านตาราง cmprocessing (สำหรับงานประมวลผลชุดใหญ่/ปิดงวด):

ชั้น เทคโนโลยี หน้าที่ ไฟล์หลักที่พบ
1. UI / Validation ASP.NET WebForms (C#) รับ input, validate เบื้องต้น, แสดงผล, เรียกชั้นถัดไป GCOOP/Saving/Applications/loan/*.aspx.cs (CORE 142 หน้า, AERO 29 หน้า)
2. Batch Processing PowerBuilder 12.5 (PBProcess.exe) งานที่ประมวลผลเป็นชุด/ใช้เวลานาน เช่น ปิดเดือน ตั้งยอดค้าง คำนวณดอกเบี้ยคืน GCOOP/PBProcess/pcloan.pbl
3. Service Layer PowerBuilder 12.5 (WCF-hosted) ธุรกรรมที่ต้องผลตอบกลับทันที เช่น บันทึกใบเสร็จ อนุมัติคำขอ คำนวณดอกเบี้ยรายตัว GCOOP/PBService125/loansrv.pbl, lncoopsrv.pbl

สถาปัตยกรรม CORE vs AERO: โปรเจกต์นี้มี source 2 ชุดคู่ขนาน — CORE (baseline เต็มรูปแบบ) และ AERO (fork ที่ปรับแต่งเฉพาะลูกค้า) จากการตรวจโค้ดจริงใน CoreSavingLibrary/MenuBar.cs (บรรทัด 135, 173, 211) พบว่าระบบตัดสินใจว่าจะพาผู้ใช้ไปหน้าเวอร์ชันไหน โดยอ่านค่าคอลัมน์ amsecwins.core_flag ต่อหน้าจอ (ต่อ win_parameter):

// CoreSavingLibrary/MenuBar.cs (บรรทัด 211, ย่อ)
(case b.open_type when 3 then savUrl3 else ( case b.core_flag when 1 then savUrl0 else savUrl1 end ) end)
|| '/GCOOP/Saving/Applications/' || ...

กล่าวคือ core_flag = 1 → ใช้เส้นทาง CORE (savUrl0) / ค่าอื่น (เช่น 0) → ใช้เส้นทาง AERO/extend (savUrl1) ตรงตามกติกาที่ผู้ใช้ระบุ

⚠️ ข้อจำกัดของการวิเคราะห์นี้: ค่าจริงของ amsecwins.core_flag เป็น ข้อมูล runtime ที่เก็บในฐานข้อมูลของแต่ละสหกรณ์ (ตั้งค่าได้ต่อสหกรณ์ ต่อหน้าจอ) ไม่ได้อยู่ใน source code หรือไฟล์ export ที่มีอยู่ในเครื่องนี้ (ไฟล์ Save-Rows-As ของ amsecwins ที่พบมีเฉพาะโมดูล admin/sms เท่านั้น ไม่มี entry ของ loan) จึงไม่สามารถยืนยันค่า core_flag จริงของแต่ละหน้าได้ 100% ในเอกสารนี้จึงใช้หลักฐานทางอ้อมที่ตรวจสอบได้จริงแทน คือ การ diff เนื้อหาไฟล์ CORE กับ AERO ทีละหน้า

กลุ่ม จำนวนหน้า การตีความ
หน้าที่มีเฉพาะใน CORE 118 ไม่มี override ใน AERO →ทำงานจาก CORE เพียงทางเดียว (ไม่ต้องพึ่ง core_flag)
หน้าที่มีทั้งใน CORE และ AERO และ เนื้อโค้ดต่างกันจริง 21 มีการ fork/ปรับแต่งจริง — ระบบต้องใช้ core_flag ตัดสินว่าจะรันตัวไหน (เอกสารนี้สันนิษฐานว่า deployment สาย AERO ตั้ง core_flag ≠ 1 ให้กลุ่มนี้ใช้เวอร์ชัน AERO ที่ปรับแต่งแล้ว)
หน้าที่มีทั้งใน CORE และ AERO แต่ เนื้อโค้ดเหมือนกันทุกบรรทัด 1 (ws_lon_cls_process_setmb_npl_ctrl) ไฟล์ AERO เป็นสำเนา CORE เป๊ะ (diff = 0 บรรทัด) ไม่ว่า core_flag จะเป็นค่าใดผลลัพธ์เหมือนกัน
หน้าที่มีเฉพาะใน AERO (ฟีเจอร์ใหม่ที่ CORE ไม่มี) 7 ไม่มี CORE ให้ fallback → ต้องรันจาก AERO เท่านั้น
รวมหน้าจอทั้งหมดในโมดูล loan 146 หน้า (ไม่ซ้ำ) 118 + 21 + 1 + 7 - บวกโฟลเดอร์ระบบ dlg (dialog ย่อย ใช้ร่วม)

3. ขั้นตอนการประมวลผลอย่างละเอียด (Detailed Processing Steps)

3.1 วงจรคำขอกู้ → อนุมัติ → เบิกจ่าย (Request → Approval → Disbursement)

  1. สมาชิกยื่นคำขอ — ws_lon_reqloan_ctrl (PostMemberNo) ไล่ตรวจสิทธิ์ 14 เงื่อนไข (สถานะสมาชิก/ใบคำขอค้าง/ผิดนัดชำระ/อายุ-งวดหุ้น/วงเงินรวม MAXALLLOAN_AMOUNT) → เรียก wcf.NLoan.of_getcollpermiss คำนวณวงเงินค้ำ + LoanMasterClass.CalLoanCredit() (ฝั่ง .NET ล้วน ไม่ผ่าน WCF) → บันทึก LNREQLOAN (status รออนุมัติ)
  2. เจ้าหน้าที่อนุมัติ — ws_lon_apvloan_ctrl (SaveWebSheet) รวมแถวที่เลือกส่งเข้า wcf.NLoan.of_saveapv_lnreq ครั้งเดียวทั้งชุด → of_gennewcontractno ออกเลขสัญญา (coop 056001 มี flow เสริม 3 ขั้น: อัปเดตดอกเบี้ยหักกลับ + สร้างคำขอเปลี่ยนแผนประกัน + พิมพ์ใบสั่งจ่าย PDF เฉพาะสหกรณ์นี้)
  3. เบิกจ่ายจริง — ws_lon_post_loan_receive_ctrl (LoopSave) ดึงรายการที่ยังไม่ผ่านรายการจาก slslippayout → เรียก wcf.NLoan.of_saveslip_postlnrcv (WCF sync) → update slslippayout.lotpay_seqno + lnreqloan.loanrequest_status=1
  4. [ทางเลือก] พิมพ์/ค้นหาก่อนเบิก — ws_lon_rcvloan_ctrl เป็นหน้าค้นหา+พิมพ์เท่านั้น (ไม่มีการบันทึก) การบันทึกจริงถ้าผ่านหน้านี้เกิดที่ dialog แยก dlg/wd_lon_rcvlon_ctrl

3.2 วงจรรับชำระที่เคาน์เตอร์ (Payment)

  1. ws_lon_payment_ctrl (PostMemberNo) ตรวจ token กันทำซ้ำข้ามเครื่อง → ตรวจค้างชำระผ่าน InitLnRcv()/InitLnRcvKeepArrear() → คำนวณค่าปรับผิดนัดอัตโนมัติผ่าน of_setloanfilepayment → คำนวณดอกเบี้ยใหม่ผ่าน ReCalint (เช็ควันทำการ)
  2. SaveWebSheet เรียก wcf.NLoan.of_saveslip_payin (synchronous, XML 5 ส่วน: head/share/loan/etc/expense) แบบ call-and-wait
  3. หลังบันทึกสำเร็จ ยิงต่อทันทีด้วย WebUtil.runProcessingFin(state) (asynchronous) เพื่อประมวลผลด้านการเงินที่ไม่ต้องรอผล — เป็นตัวอย่างเดียวในระบบที่ธุรกรรมเดียวใช้ ทั้ง sync และ async ต่อกัน
  4. เงื่อนไขพิเศษ: coop 080001 ต้องเปิดวันทำการระบบการเงินก่อน, coop 050001 ห้ามแก้ยอดดอกเบี้ยที่ชำระ, AERO เพิ่มการตรวจสิทธิ์ซื้อหุ้นพิเศษ (CheckBuySPX)

3.3 วงจรปิดงวด/ตั้งยอดค้าง/NPL (Period-close & Arrear)

  1. ws_lon_clsmth_ctrl (Cls_Mth_Close) → ยิง batch คู่กัน 2 งาน: LNCLOSEMONTH (→ n_cst_proc_slcls.of_slcls_mth()) และ LNCALINTARRNEW (→ n_cst_proc_slcls.of_slcls_mth_genintarr_aero() — เวอร์ชันเฉพาะ AERO)
  2. ws_lon_cls_process_setmb_npl_ctrl (JsPostProcessNPL) → ยิง batch PROCSETARRMTHNPL (→ n_cst_sl_mth_arrear_process.of_initservice() + of_procnpl()) — ตั้งยอดค้างชำระ/สถานะ NPL ของสมาชิกทั้งหมดประจำงวด (หมายเหตุ: ฟังก์ชัน OfCheckUseProcess ที่ควรกันรันซ้ำถูกเตรียมไว้แต่ยังไม่ถูกเรียกใช้จริงในโค้ด — ความเสี่ยง)
  3. ws_lon_proc_prinarrear_aero_ctrl (เฉพาะ AERO) คำนวณเบี้ยปรับเงินต้นค้างแบบ synchronous ล้วน (ไม่ผ่าน batch/WCF) ด้วย finerate hardcode 3.25% บันทึกเข้าโมดูล "เก็บเงินอื่นๆ" (kprcvkeepother)
  4. ws_lon_auditloan_ctrl เป็นเครื่องมือแก้ไขค่า field สัญญาโดยตรงหลังตรวจสอบ (ไม่ใช่แค่รายงาน) — เฉพาะ coop 017001 เท่านั้นที่บันทึก audit log ทุกครั้งที่แก้ไข (lnauditloanlog)

7. สรุปข้อจำกัดของการวิเคราะห์ (Methodology Note)

  • การวิเคราะห์นี้อ่านจาก source code ที่มีอยู่ในเครื่อง (ASP.NET .aspx.cs แบบ uncompiled อ่านได้เต็ม 100%, PowerBuilder .pbl แบบ compiled ใช้ strings -e l แกะ UTF-16LE ซึ่งอาจมีบาง function body คลาดเคลื่อนจากขอบเขต object ที่ประกาศไว้ เหมือนที่เคยพบใน pckeeping.pbl)
  • ค่า amsecwins.core_flag จริงเป็น runtime configuration ต่อสหกรณ์ ไม่มีอยู่ใน source tree ที่ตรวจสอบได้ในเซสชันนี้ — ข้อสรุปเรื่อง "หน้าไหนใช้ CORE หรือ AERO" ในเอกสารนี้อ้างอิงจาก การมีอยู่ของไฟล์และผลต่างของเนื้อโค้ด เป็นหลักฐานทางอ้อม ไม่ใช่ค่า core_flag ที่ยืนยันจากฐานข้อมูลจริง
  • หน้าจอ CORE-only 118 หน้าในหัวข้อ 2.1.2 จัดหมวดจาก naming convention เป็นหลัก (🟡) ยังไม่ได้อ่าน source ทีละบรรทัดครบทุกหน้า — สามารถขอเจาะลึกเพิ่มเติมเป็นรายหน้าหรือรายหมวดได้ในคำขอถัดไป
